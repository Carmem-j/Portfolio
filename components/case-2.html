1. Contexto Estratégico

Problema identificado

Um widget de gráficos temporais que estava sendo usado nos ambientes de clientes de fracking...

Objetivos

  • Integração de features de visualização de dados para o principal widget no-code de dados temporais da plataforma, pois o widget de referência é híbrido e alguns recursos apenas são possíveis via código;
  • Substituição completa do widget usado atualmente, pela complexidade técnica e alto custo de manutenção e correção de bugs;
  • Permitir o uso das features originárias do widget usado em fracking para outros mercados.

2. Desafio

A construção do mapa de stakeholders no início do processo foi essencial para identificar todos os atores importantes do projeto e agrupá-los para uma melhor gestão das partes interessadas, de acordo com a fase do projeto em relação à tríade: tecnologia, negócios ou usuário.

O desafio maior deste projeto esteve presente no trabalho de mediação entre os três pilares citados, principalmente pelo fato de que a plataforma atual não suportava uma “estrutura de dados ideal” e por conta disso um dos principais gaps do projeto era técnico. Sendo assim, foi necessário um trabalho cuidadoso de priorização, de forma que fosse possível fazer pequenas entregas em fases para adaptação do usuário na transferência de uso do novo widget, mas de forma que as primeiras entregas atendessem a requisitos de viabilidade técnica para o momento e atendessem às necessidades mais urgentes dos usuários.

O segundo maior desafio foi inserir no discovery uma etapa de aprendizado de domínio, pois até o momento eu projetava soluções de visualização de dados genéricas para qualquer mercado ou para o domínio de drilling, pertencente ao mercado de O&G. No entanto, o projeto se tratava de uma iniciativa de trazer soluções também para o domínio de fracking, que se trata de outro método de extração de petróleo, portanto era necessário um estudo extenso sobre processos, jornadas dos diferentes perfil de usuários e sobre a estrutura de dados, que era completamente diferente. Para isso, o uso de IA generativa foi essencial, pois além de me auxiliar no processo de UX research e imersão no novo domínio que eu ainda não tinha conhecimento, o uso dela foi importante para otimizar o processo de documentação de transcrições de reuniões e organização de demais documentos.

3. Decisões Estratégicas

O ponto forte da entrega do projeto, que ainda está em andamento, foi sua priorização estruturada e acordada por todas as partes, respeitando o ritmo de adaptação da plataforma tecnicamente para suportar uma estrutura de dados não suportada, algo negociado cross-squads, mas levando em consideração a necessidade do usuário. Sendo assim, a divisão de entregas teve a seguinte divisão, depois de sessões de priorização em que facilitei usando o framework MoSCoW com diferentes grupos de stakeholders:

  • Primeira entrega: features consideradas must have, que eram viáveis tecnicamente no momento e podiam ser utilizadas por clientes de outros domínios;
  • Segunda entrega: features consideradas must have, que eram viáveis tecnicamente no momento e atendiam somente ao domínio de fracking;
  • Terceira entrega em diante ainda seriam negociadas;

4. O Impacto

Além dos ganhos da substituição de um widget caro para correção e manutenção, com o uso do widget no-code de dados temporais os usuários ganham autonomia para criação e edição de visualizações de dados e o mais importante: não é mais necessário solicitar auxílio de um cientista de dados somente para realizar edições pontuais via código, tendo em vista que o widget substituído utiliza uma linguagem de programação interna da empresa.

Outro ganho se dá pelo fato do widget no-code poder ser sincronizado com os demais tipos de widgets de dados temporais existentes na plataforma. Sendo assim, os usuários de fracking ganham com uma gama de ferramentas de visualização, análise e de navegação individuais ou agrupadas para usar no dashboard da plataforma, que é compatível com o widget no-code que recebeu as features.

Vale mencionar também o esforço em tornar a solução genérica o suficiente para atender não somente aos clientes de fracking, mas também aos clientes de drilling e outros mercados, o que expande o uso das features em diferentes frentes. Isso foi graças a muitas entrevistas e reuniões de alinhamento e validação e, com essa iniciativa, foi possível encontrar um caso de uso de grande aderência dentro de um produto interno de drilling, no mapeamento de KPIs, em que uma das features projetadas traria uma redução de espaço vertical de tela de rolagem de aproximadamente 40%.

Desta forma, considerando que os clientes de drilling atualmente correspondem à maior fonte de receita da empresa, seguidos de fracking, o maior ganho do projeto, além do oferecimento de autonomia e uma gama de features para uso do dashboard no dia a dia, está também ao escalar o uso das features mapeadas e projetadas para usuários dos clientes das duas maiores fontes de receita da companhia.
